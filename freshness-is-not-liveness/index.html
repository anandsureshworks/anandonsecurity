Freshness ≠ liveness

Your monitor says the agent is fine. It’s been dead for three weeks.

Anand Suresh · 2026-06-26

Fresh output over a dead engine is the most boring catastrophe in autonomous systems: no crash, no stack trace, no page at 3 a.m. Just a green dashboard and a quiet, expensive lie.

I know because I shipped it. More than once.

The dashboard is green. The output file has today’s timestamp. The scheduled job exited 0. Every check you have says the system is alive — and it has been dead for weeks. The fix is never smarter prediction. It’s the discipline to detect the absence of work.

Predict less, detect more. You can’t forecast the shape of a silence you’ve never seen. You can always detect the one thing that’s true of every dead engine: it stops producing.

The one word your monitor is missing

Freshness asks one question: was this written recently? Liveness asks a strictly stronger one: is the thing that writes it actually working?

A fresh file over a dead producer passes the first and fails the second. Every silent failure I’ve shipped has lived in that gap.

Liveness is four invariants, not one: the output exists, it’s fresh, it’s non-empty, and — the one everybody skips — it’s producing.

“Producing” means the content actually advanced: the counter moved, the baseline filled, the analysis is real and not a placeholder sitting behind a recent date. It’s the only invariant that separates a working engine from a fresh corpse — and the only one freshness can’t see.

The mechanism is mundane

You have an agent on a scheduler — launchd, cron, a Kubernetes CronJob. It wakes, does its work, writes a file, exits. Every scheduled run is a fresh process. A counter you initialize in __init__ and hold in memory resets to its starting value on every run, because the process that held it died and a new one started from scratch. The adaptive logic meant to engage after N runs never engages — N is always 1. The code reads correctly. Your tests pass, because they run in one long-lived interpreter that hides the bug. In production the work never advances, and nothing tells you, because the file keeps getting written with a fresh timestamp.

I have watched a real one: an analysis whose baseline sat permanently at not_ready behind a perfectly recent generated_at. New file every cycle; the thing inside it never moved. Freshness monitoring reported green for weeks.

It caught its own author

The part that should bother you: I write reliability tooling, and I fell for this inside the tool that scans for it. A secrets-scan of mine once reported clean — fresh output, recent run — while it had quietly stopped doing the actual scanning.

The bug was trivial. The real failure was a habit of mind: trusting a system’s “reliability” — a green check, a recent run — instead of reasoning from first principles about what the check actually proves. A timestamp proves a write happened. It proves nothing about the work behind it. I knew that, and trusted the green anyway. That is the whole trap, and it does not care how well you understand it.

What to actually do

Three moves, and you can do all of them by hand, no library required:

  1. Make cross-run state durable by construction. If it must survive a restart, it must live on disk, written atomically (temp file + rename, never in place). And test it by killing the process — spawn real subprocesses that increment, save, and exit, and assert the value climbs 0→1→2→3 across genuine process death. A test that reuses an object in one interpreter passes with the bug fully present.
  2. Replace freshness checks with a liveness contract. Stop asking “recent?” Ask “exists, fresh, non-empty, and producing?” Define producing as a predicate over the payload — the counter advanced, the field filled — and fail the check when it’s false, even when the timestamp is immaculate.
  3. Fail loud — but only on transitions. A detector with no response channel is a sensor wired to nothing. Route the verdict somewhere you’ll actually see it, but fire on the change, not every tick, or you’ll train yourself to mute it. And make certain the alarm can never crash the run it’s protecting.

I packaged these three into a small, dependency-free harness — agentliveness — because I got tired of re-learning the same lesson. But the harness is incidental. The discipline is the point.

A dead engine doesn’t scream. It goes quiet behind a fresh timestamp and waits for you to notice. So stop trusting the timestamp — and build the thing that notices.
